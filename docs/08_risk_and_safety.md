# 08 — 安全边界与风险声明

## 研究边界

本项目的所有分析遵循以下边界：

1. **只读** — 不修改设备文件系统、配置、固件
2. **离线** — 不连接设备发送命令或 payload
3. **不刷机** — 不通过 USB/UART/OTA 向设备刷写数据
4. **不爆破** — 不对设备进行密码猜测或枚举
5. **不拆机** — 不进行物理攻击（UART、JTAG、eMMC dump）

## 不推荐的操作

以下是明确**不推荐**的操作，可能造成设备变砖或数据丢失：

### 1. 刷写 modified OTA
- **风险**：签名验证失败可能导致更新中断，设备进入不可用状态
- **原因**：RSA 签名无法绕过，修改后的 OTA 无法通过验证
- **即使**签名验证被绕过，错误的固件修改可能导致设备无法启动

### 2. 密码爆破
- **风险**：无效操作，浪费时间
- **原因**：SHA256 哈希强度 + 3 次限制，爆破不可行
- **设备可能有反爆破机制**（如锁定周期）

### 3. WebSocket Blind Fuzz
- **风险**：可能导致服务崩溃或设备重启
- **原因**：9307 处于 stub 状态，对未实现方法的未知响应可能触发 bug
- **非必要不应向生产设备发送未知数据**

### 4. UART / 拆机 / 硬改
- **风险**：物理损坏、保修作废、设备变砖
- **前提**：需要相应的硬件工具和救砖能力
- **不建议在没有完整救砖方案的情况下进行**

## 分析过程中的安全实践

- 固件下载使用 `curl`/`wget`，仅读取官方 CDN 内容
- ext4 解析使用只读挂载或 Python 解析，不修改提取内容
- 密码哈希验证使用独立 Docker 容器，不触及本地文件系统
- 设备连接仅限 `adb connect` 和 `nc -vz` 端口确认

## 如果继续研究

### 建议优先做的事

1. 寻找合法途径获取完整 rootfs（官方固件下载、社区共享）
2. 提取并分析 `adb_auth.sh` 完整内容（纯文本分析）
3. 提取并分析 `safepowerdown` 文件内容
4. 解压 JS bundle 文件了解 WebSocket 调用格式
5. 在 GitHub 上搜索相关研究，联系 PenUniverse 社区

### 不建议做的事

- 向设备发送任何形式的 payload
- 尝试绕过签名验证刷写固件
- 对设备进行端口扫描以外的主动探测
- 发布可用于攻击的工具或脚本
